Схемы прохождения транзакций через L1 (Arbitrum) → L2 (PBFT) → L3 (Disruptor/Execution)
Пользователь / Кошелёк L3 — Disruptor / StateExecutionHandler L2 — PBFT консенсус (3 валидатора) L1 — Arbitrum Bridge (расчёты) Успешная операция Отклонено / Ликвидировано
1. Внесение средств L1 → L2
Ты отправляешь USDC на bridge-контракт Arbitrum (L1). Валидаторы через Poller подхватывают выкупленный тикет и проводят через консенсус. После коммита — бухгалтер зачисляет на L2 счёт. 
flowchart TD
    A["👤 Ты (кошелёк L1)"] -->|"POST /api/deposit
{userId, amount}"| B["Валидатор (REST API)"]
    B -->|"depositL1()
createRetryableTicket()"| C["ArbitrumBridge
L1"]
    C -->|"ждёт ticketRedeemDelay
(~2s симуляция)"| D["processTickets()
→ auto-redeem"]
    D -->|"tx DEPOSIT → outbox"| E["outbox[]"]
    E -->|"ArbitrumBridgePoller
(каждую 1с)"| F["Mempool"]
    F -->|"PBFT: PrePrepare→Prepare→Commit
2/3 валидаторов"| G["StateExecutionHandler
L3 Disruptor"]
    G -->|"dedup по ticketId"| H{"Уже обработан?"}
    H -->|"да"| I["⛔ SKIP"]
    H -->|"нет"| J["marginManager.deposit()
freeBalance += amount"]
    J --> K["✅ Баланс L2: +500 USDC"]

2. Открытие позиции L3 — исполнение
Ты ставишь лимитный ордер. Валидаторы голосуют, Disruptor проверяет маржу, OrderBook ищет встречную заявку. При совпадении — сделка, бухгалтер создаёт позицию. 
flowchart TD
    A["👤 Ты"] -->|"POST /api/order
{isBuy, price, amount, leverage}"| B["Mempool"]

    B -->|"PBFT consensus
2/3 vote"| C["StateExecutionHandler
L3 Disruptor"]

    C -->|"tx.toOrder()"| D{"checkAndLockMargin()
freeBalance ≥ margin?"}

    D -->|"❌ Нет"| E["⛔ ORDER REJECTED
Insufficient funds"]
    D -->|"✅ Да"| F["freeBalance -= margin
lockedMargin += margin"]

    F --> G["OrderBook.processOrder()
price-time priority"]

    G --> H{"Есть встречная заявка
по crossing price?"}

    H -->|"❌ Нет"| I["Заявка висит в стакане
ждёт встречного ордера"]

    H -->|"✅ Да"| J["MATCH → Trade[]
{buyer, seller, price, qty}"]

    J --> K["marginManager.processTrade()
→ комиссии"]
    K --> L["updatePositionAfterTrade()
создать/увеличить позицию"]

    L --> M["✅ Позиция открыта
Size, EntryPrice, Margin, Leverage"]

3. Маржа-контроль и ликвидация каждое обновление цены
Каждый раз, когда оракул даёт новую цену — валидаторы проводят UPDATE_ORACLE через консенсус. Disruptor пересчитывает ликвидационные цены для каждой открытой позиции. Если маржа ниже порога — принудительное закрытие. 
flowchart TD
    A["📡 Oracle Price Feed"] -->|"UPDATE_ORACLE tx"| B["PBFT → Commit"]

    B --> C["StateExecutionHandler
L3 Disruptor"]

    C --> D["OracleService.setPrice(marketId, price)"]

    D --> E["LiquidationEngine
.checkLiquidations(marketId)"]

    E --> F["Для каждого пользователя
с позицией в этом marketId"]

    F --> G{"Режим маржи?"}

    G -->|"Изолированная"| H["Расчёт: liquidationPrice =
entryPrice ± (margin / size / maint%)"]
    G -->|"Кросс"| I["Расчёт: totalEquity =
freeBalance + Σ(margin + unrealizedPnL)"]

    H --> J{"Цена пересекла
liquidationPrice?"}
    J -->|"❌ Нет"| K["✅ Позиция в безопасности"]
    J -->|"✅ Да"| L["🔴 Forfeit margin → close position
Убыток = вся маржа"]

    I --> M{"Equity < Σ(size×price×maint%) ?"}
    M -->|"❌ Нет"| N["✅ Портфель в порядке"]
    M -->|"✅ Да"| O["🔴 Liquidate ALL cross positions
Все маржи сгорают, freeBalance = 0"]

    L --> P["FundingCalculator
.applyFunding()
(платежи long→short)"]
    O --> P

4. Закрытие позиции L3 — PnL realisation
Чтобы закрыть позицию — ставишь встречный ордер. Disruptor проверяет: если ордер уменьшает существующую позицию — новая маржа не нужна. OrderBook ищет ликвидность, после сделки бухгалтер закрывает позицию и размораживает剩余 маржу. 
flowchart TD
    A["👤 Ты"] -->|"POST /api/order
(противоположная сторона)"| B["Mempool → PBFT"]

    B --> C["StateExecutionHandler
L3 Disruptor"]

    C --> D["checkAndLockMargin()
0 — reducing position
(новая маржа не нужна)"]

    D --> E["OrderBook.processOrder()
→ match по рынку/лимиту"]

    E --> F["Trade[] исполнена"]

    F --> G["updatePositionAfterTrade()"]

    G --> H{"Что делает сделка
с существующей позицией?"}

    H -->|"Частичное закрытие"| I["PnL = (exitPrice − entryPrice) × qty
+ Прибыль → freeBalance
+ Пропорциональная маржа разморожена"]
    H -->|"Полное закрытие"| J["Весь PnL реализован
Вся маржа разморожена
Позиция = 0"]
    H -->|"Переворот (over-close)"| K["Закрыть старую позицию
Открыть новую в opposite direction
с остаточным объёмом"]

    I --> L["✅ Позиция закрыта / уменьшена"]
    J --> L
    K --> L

5. Вывод средств L2 → L1
Ты подписываешь запрос RSA-ключом. Валидаторы проверяют подпись и баланс через консенсус. После коммита — средства списываются с L2 и ставятся в очередь PendingWithdrawal. Финалайзер отправляет запрос в ArbitrumBridge. Через challenge window (7 дней / 30с) — деньги на L1. 
flowchart TD
    A["👤 Ты"] -->|"POST /api/withdraw
{userId, amount, RSA-signature}"| B["Mempool → PBFT"]

    B --> C["StateExecutionHandler
L3 Disruptor"]

    C --> D["verify RSA signature
DexSignatureUtil.verify()"]

    D --> E{"Подпись валидна?"}
    E -->|"❌ Нет"| F["⛔ WITHDRAW REJECTED
Invalid signature"]
    E -->|"✅ Да"| G["marginManager.withdraw()"]

    G --> H{"freeBalance ≥ amount?"}
    H -->|"❌ Нет"| I["⛔ WITHDRAW REJECTED
Insufficient funds"]
    H -->|"✅ Да"| J["freeBalance -= amount
→ PendingWithdrawal[]"]

    J --> K["WithdrawalFinalizer
(каждые 3с)"]
    K -->|"bridge.initiateWithdrawal()"| L["ArbitrumBridge
→ WithdrawalRequest"]

    L --> M["ждёт challengeWindow
~7 дней (30с в dev)"]

    M --> N{"Кто-то
зачелленджил?"}
    N -->|"✅ Да challengeWithdrawal()"| O["⛔ WITHDRAW REJECTED
Challenged — средства заблокированы"]
    N -->|"❌ Нет"| P["bridge.finalizeWithdrawals()
L1 balance += amount"]

    P --> Q["✅ USDC на L1 кошельке"]
